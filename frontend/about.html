What this is

spec2program is a prototype that turns a structured vehicle specification into a validated commissioning program — the ordered sequence of diagnostic actions (open a session, unlock security access, flash software, write configuration, validate, clear fault codes) that brings a vehicle's electronics into a defined, tested state at the end of the production line.

It exists to explore one specific idea: today, programs for vehicle commissioning are created from specifications that pass through many organisational and technical hand-offs. Every hand-off is a place where errors creep in, and those errors directly cost cycle time, production stability, and commissioning quality. This tool asks whether an AI model, constrained by an explicit rule-based safety check, can remove some of that risk automatically.

How it works

  1. Input — the specification. A JSON document bundling the heterogeneous sources that actually drive commissioning: the bill of materials (which ECUs are present, their part numbers and current/target software versions), the vehicle configuration (drivetrain, market, options), which UDS (Unified Diagnostic Services, ISO 14229) each ECU supports, and the process standards that must be honoured (e.g. "security access must precede any flash").
  2. Generate — the program. The specification is handed to an LLM (or, with no API key configured, a deterministic offline rule-based planner) with instructions to produce an ordered program: which UDS action to take on which ECU, and in what order, respecting the safety and dependency rules above.
  3. Validate — never trust blindly. The generated program is checked against the original specification before it's shown as "passed": every ECU and UDS service referenced must actually exist and be supported, security access must come before any flash or parameter write on the same ECU, step dependencies must resolve, and every ECU that needs a software update must actually get flashed. If the model's output fails any of this, the failure is shown explicitly — errors and warnings, not a silent pass.
  4. Analyse — where's the optimisation potential. The validated program is summarised into concrete numbers: total steps, estimated cycle time, how many of the vehicle's ECUs were touched, and how many steps have no dependency on each other (a proxy for what could run in parallel on multi-channel tester hardware).

Resilience

A real LLM call can fail for reasons outside anyone's control — a network hiccup, a rate limit, a model that returns malformed JSON. Rather than surfacing a hard error, the pipeline falls back to the offline rule-based planner automatically and says so plainly in the result — the product is designed to degrade visibly, never silently.

What it is not

This is a prototype built to explore the problem shape, not a production commissioning system. The sample vehicle data (ECUs, part numbers, UDS service lists) is realistic but invented — it is not real manufacturer data. The rule-based validator encodes a small, illustrative set of safety and structural checks, not a complete specification of real commissioning process standards.

Stack

Python, FastAPI backend; a dependency-free HTML/CSS/JS frontend served by the same backend so the whole thing ships as one deployable container; pluggable LLM backend (offline mock, or any OpenAI-compatible endpoint — currently wired to OpenRouter); pytest test suite; Docker + Docker Compose for deployment.
